## Workstream 1 — Redirect SSRF hardening

### Problem
`server/lib/ai-providers/openai-compat.ts` validates `cfg.baseURL`, then calls `fetch()` with default redirect following. A public base URL can redirect to private/metadata hosts after the initial check.

### Design
Add a shared helper in `openai-compat.ts` (or tiny adjacent helper in same file to avoid new module churn):

```ts
fetchOpenAICompat(url, init, { maxRedirects = 3 })
```

Behavior:
1. Always `redirect: 'manual'`
2. Reuse existing `assertFetchTargetSafe(url)` before every hop (including initial)
3. On 3xx:
   - Require `Location`
   - Resolve absolute URL via `new URL(location, currentUrl)`
   - Reject non-http(s), userinfo, private/loopback/metadata via `ssrfUnsafeUrlReason`
   - **Do not forward `Authorization` if host/port/scheme changes** (same-origin only credential forwarding)
   - Cap hops at 3; throw clear error beyond that
4. Return final non-redirect `Response` to existing parsers

Wire into all four call sites:
- LLM chat completions
- embeddings
- image generations
- TTS speech

Keep existing production-only / `AI_ALLOW_PRIVATE_NETWORK_TARGETS` semantics of `assertFetchTargetSafe`.

### Tests
Extend `tests/routes/ingress-security.test.ts` and/or `tests/core/core-01.test.ts` provider suite with local `http.createServer` redirect chains:
- public-looking initial URL → redirect to `127.0.0.1` → reject
- redirect to `169.254.169.254` → reject
- same-host redirect preserves Authorization and succeeds
- cross-host same-public redirect strips Authorization (or rejects if auth required)
- redirect loop / >3 hops rejects
- missing `Location` rejects

Reuse the existing loopback server pattern already used by `makeOpenAICompatLLMProvider` tests in `tests/core/core-01.test.ts`.

### Residual gap (documented, not fixed this pass)
Literal-form SSRF still cannot fully close DNS rebinding. Leave a short comment near the helper pointing at the existing residual-gap note in `validation.ts`.

### Design

   - Hard (must pass): `identity`, `whitespace_reflow`, `rename_character`, `scene_shuffle`, `scene_reverse`, `scene_dup_padding`
   - Known-failing witness: `empty_verbosity` only
2. Runner behavior:

3. Add package script:

Prefer keeping the runner executable and adding a small unit test for the pass/fail classification pure functions extracted from it.

## Workstream 3 — Multi-file upload race

### Problem
`StoryConfigForm.processFiles` closes over stale `uploadedFiles` and overwrites concurrent `FileReader` completions.

### Design
Minimal root-cause fix in `src/components/startscreen/StoryConfigForm.tsx`:
1. Read selected batch with `Promise.all` of `FileReader`/or `file.text()`
2. Call parent once with functional update if possible
3. Prefer changing parent callback to accept either:
   - `UploadedFile[]`, or
   - keep prop as array and use `onUploadedFilesChange([...uploadedFiles, ...batch])` only after the full batch is ready

Best minimal API change:
- Keep `onUploadedFilesChange: (files: UploadedFile[]) => void`
- In processFiles, await all reads, then `onUploadedFilesChange([...uploadedFiles, ...newFiles])` once
- Also change remove/category to stay correct
- Mirror limits/accepts with StartScreen drop path where cheap:
  - same max size (2MB)
  - same extensions, or document intentional divergence

Also fix remove/update if they still close over stale arrays when rapid multi-ops occur; use functional parent updates if StartScreen can accept them without broad API churn. If StartScreen only has `setUploadedFiles`, simplest robust fix is:

```ts
onUploadedFilesChange can stay, but processFiles should use:
set via a batch callback from parent:
onUploadedFilesChange with functional form optional
```

Preferred clean approach with minimal surface change:
- Change prop type to:
  `onUploadedFilesChange: (update: UploadedFile[] | ((prev: UploadedFile[]) => UploadedFile[])) => void`
- Parent passes `setUploadedFiles`
- All child mutations use functional updates

Align `processFiles` size/ext filters with `StartScreen` drop filters.

### Tests
No current frontend unit harness for React. Prefer:
- small pure helper extract `readUploadedFiles(files): Promise<UploadedFile[]>` + node test if easy
- or a focused pure-function test of batch accumulation logic extracted from the component

If extracting is too invasive, document manual QA steps and add a pure batch-merge helper with tests.

## Workstream 4 — ScriptIDE autosave + restore correctness

### Problem
1. `setInterval(..., 30_000)` is recreated on every edit dependency change → continuous typing can starve server saves
2. Restore chooses longer draft, not newer draft
3. Saving indicator does not reflect durable success/failure

### Design

#### 4a. Stable autosave
In `src/components/ScriptIDE.tsx`:
- Keep always-current refs for `scriptText`, `snapshots`, `characters`, `researchNotes`, `isDarkMode`
- One mount-only interval (or trailing debounce + interval) that reads refs
- Also flush on:
  - `visibilitychange` → hidden
  - unmount
- Keep localStorage 500ms debounce

#### 4b. Local revision timestamps
- Persist `script_draft_updated_at` alongside `script_draft`
- On every successful local write, set `Date.now()`
- Server already stores `updated_at` in Stage and returns `updatedAt`

#### 4c. Restore conflict rule
On `/api/scriptide/load`:
```
if server empty → keep local
if local empty → take server
if both present:
  prefer higher updatedAt
  if equal timestamps, prefer longer as tie-break
```
Never overwrite a newer local draft solely because server text is longer.

#### 4d. Honest save status
Replace fake 800ms “Saving…” timer with states:
- `idle | saving-local | saved-local | saving-server | saved-server | save-failed`
Show compact status near existing indicator:
- `Saved locally`
- `Saved to server`
- `Save failed` (local quota / server error)

Do not claim server save success unless `/api/scriptide/save` returns ok.

#### 4e. Backend
No schema migration required if client uses existing `updatedAt`. Optionally accept client `clientUpdatedAt` on save for future conflict detection; **not required** if server always stamps save time and client compares timestamps correctly.

Extend `tests/routes/scriptide.test.ts`:
- save then load roundtrip preserves text + updatedAt
- load returns updatedAt for non-empty state
