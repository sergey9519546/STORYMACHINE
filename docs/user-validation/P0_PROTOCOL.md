# P0 Validation Protocol — Story Graph Enhanced Diagnostics

**Date:** 2026-07-15  
**Purpose:** Test Phase 2 Enhanced Diagnostics with real screenwriters  
**Duration:** 1 week (5-10 sessions)  
**Gate:** GO/NO-GO decision on building Phase 3-4

---

## Objective

Answer the core question from ROADMAP.md:
> "Does this make you want to run your own draft?"

Specifically for Story Graph:
> "Do these structural diagnostics help you improve your screenplay?"

---

## Who to Recruit

### Target Profile
- **Real screenwriters** with drafts in hand
- Any career tier (emerging, working, established)
- Currently revising a feature screenplay
- Open to feedback tools

### Where to Find Them
1. **Film school contacts** (MFA screenwriting programs)
2. **Screenwriting forums** (Reddit r/Screenwriting, Done Deal Pro, Simply Scripts)
3. **Twitter/LinkedIn** (screenwriting community hashtags)
4. **Coverage services** (writers who've paid for notes before)
5. **Writing groups** (local or online critique groups)

### Qualification Questions
- "Do you have a feature screenplay draft you're currently revising?" (must be YES)
- "Have you used screenplay feedback tools before?" (any answer acceptable)
- "Would you be willing to spend 30 minutes testing a new structural analysis tool?" (must be YES)

### Sample Size
- **Minimum**: 5 sessions
- **Target**: 8-10 sessions
- **Diverse**: Mix of genres, experience levels, draft stages

## Session Protocol

### Before the Session

**Materials to Prepare**:
1. Sample report with Story Graph section visible
2. Writer's script (if they share it) or use sample script
3. Screen recording setup (with permission)
4. Note-taking template ready

**Technical Setup**:
- Run Doctor on their script (or sample)
- Ensure Story Graph section renders
- Have report open and ready to share screen

### Session Flow (30 minutes)

#### 1. Introduction (3 min)
```
"Thanks for testing this. I'm building a structural analysis tool for
screenwriters and want to know if it's actually useful.

I'm going to show you a report on [your script / a sample script], and
I want to watch your reaction and hear your honest thoughts.

There are no wrong answers—brutal honesty helps me build better tools."
```

#### 2. Show Overall Doctor Report (5 min)
- Share screen with full Doctor report
- Let them scroll, don't guide
- Watch what they read, what they skip
- **Ask**: "What do you notice first?"
- **Ask**: "Does anything surprise you?"

#### 3. Focus on Story Graph Section (15 min)
- Scroll to Story Graph section
- Let them read the overall assessment
- Watch if they expand critical issues
- Watch if they read suggestions
- Watch if they look at strengths

**Key Questions**:
1. **Understanding**: "What does 'unpaid promise' mean to you?"
2. **Severity**: "Does the critical/medium/low grouping make sense?"
3. **Impact**: "Do the impact explanations help?"
4. **Suggestions**: "Would you actually try any of these fixes?"
5. **Strengths**: "Is it useful to see what's working?"
6. **Overall**: "Does this help you improve your script?"

#### 4. The Core Question (5 min)
```
"Now the most important question:

If you could run your own draft through this tool right now,
would you want to?

Why or why not?"
```

**Follow-ups**:
- "What would make you more likely to use it?"
- "What would make you less likely?"
- "What's missing?"
- "What's confusing?"

#### 5. Objections & Moments (2 min)
Ask about specific reactions you observed:
- If they seemed confused: "You paused at [X]. What were you thinking?"
- If they seemed excited: "You lit up at [Y]. What clicked?"
- If they seemed skeptical: "You frowned at [Z]. What bothered you?"

## GO/NO-GO Criteria

### GREEN LIGHT (Proceed to P1)
**If 4+ out of 5 writers show clear pull**:
- Answer "yes" to core question enthusiastically
- Take notes on suggestions
- Ask how to access the tool
- Few or resolvable objections

**Action**: Build P1 real benchmark corpus, prove discrimination

### YELLOW LIGHT (Iterate P0)
**If 2-3 out of 5 show pull, others ambiguous**:
- Mixed reactions (some love it, some don't get it)
- Common objections that can be fixed
- Value is there but presentation needs work

**Action**: Fix common objections, re-run P0 with 5 more writers

### RED LIGHT (Reframe Product)
**If < 2 out of 5 show pull**:
- Most answer "no" or "maybe" to core question
- Don't take notes or ask for access
- Fundamental objections (don't trust it, don't need it, too confusing)

**Action**: STOP. Reframe entire product. Don't build Phase 3-4.

## Success Signals

### Strong Pull (Green)
- "This is exactly what I've been looking for"
- "Can I run my script right now?"
- Writer takes notes on specific fixes
- Writer asks about pricing/access
- Writer shares with writing partner
- Writer mentions specific scenes to revise

### Weak Pull (Yellow)
- "This is interesting"
- "I'd probably try it once"
- Writer nods but doesn't take notes
- Writer has questions about accuracy
- Writer wants more features first

### No Pull (Red)
- "I don't think this would help me"
- "I don't understand what this means"
- "I don't trust automated feedback"
- Writer glazes over, doesn't engage
- Writer focuses on problems, not value

## Common Objections & Responses

### "How accurate is this?"
**Response**: "We've tested on 364 real scripts. The unpaid promise detection catches things human readers miss because it tracks every setup across the full screenplay. But you're the writer—you know if a suggestion makes sense for your story."

### "Is this replacing human feedback?"
**Response**: "No. This catches structural patterns—things that are hard to see when you're in the weeds. Think of it as a first pass that spots the big structural issues, so you can fix those before paying for coverage."

### "What if I'm breaking the rules on purpose?"
**Response**: "The suggestions are guidance, not rules. If you're breaking structure intentionally for artistic reasons, you'll know it. The tool just flags patterns that might be accidents."

### "This is overwhelming—too many issues"
**Response**: "That's why we group by severity. Start with critical issues—those are the structural problems that confuse audiences. Low-priority issues are nice-to-haves."

### "How do I know the suggestions are good?"
**Response**: "Try one. If it makes your script better, great. If not, ignore it. We show impact explanations so you can decide if the fix makes sense for your story."

## Remember

**The Goal**: Learn if writers want this, not prove they should.

**Watch reactions** > Listen to words. If they say "interesting" but don't engage, that's weak pull. If they say "I'm skeptical" but take notes on fixes, that's strong pull.

**Honest data beats validated hypotheses.** If P0 fails, that's success—you learned before wasting months on Phase 3-4.

**Demand before rigor.** Prove writers want it before proving it's accurate. An accurate tool no one wants is worthless.
